# E2E Testing Issues Log

## 2026-04-01 - Issue #1: drizzle-kit/api.js missing @electric-sql/pglite [CRITICAL]

**Test:** Infrastructure setup (project creation via CLI)
**Database:** SQLite
**Error:** `Module not found: Can't resolve '@electric-sql/pglite'`
**Screenshot:** e2e-onboarding-build-error.png

**Root Cause:**
When we moved `drizzle-kit` from devDependencies to dependencies in Phase 1, it gets bundled by Next.js. The `drizzle-kit/api.js` file internally requires ALL database drivers (pg, mysql2, better-sqlite3, @electric-sql/pglite, @libsql/client, etc.) regardless of which dialect you use. Next.js/Turbopack tries to resolve all of them at build time and fails when optional drivers aren't installed.

**Import trace:**

```
drizzle-kit/api.js
  → @revnixhq/nextly/dist/chunk-4FTADYC6.mjs (our drizzle-kit-api.ts wrapper)
  → @revnixhq/nextly/dist/index.mjs
  → route.ts (API route)
```

**Impact:** The login page still loads (the error is a build warning, not a hard crash), but it creates console errors and may cause issues in production builds.

**Suggested Fix Options:**

1. Move drizzle-kit back to devDependencies and use dynamic import with try/catch
2. Add `@electric-sql/pglite` as an optional dependency
3. Use `serverExternalPackages` in next.config.ts to exclude drizzle-kit from bundling
4. Lazy-load the drizzle-kit-api.ts module so it's only imported server-side at runtime

**Fix Applied:** Option 3 - add to serverExternalPackages in the test project's next.config.ts, then verify if this should be added to the generated template.

## 2026-04-01 - Issue #3: TableResolver chicken-and-egg problem on fresh DB [CRITICAL]

**Test:** Test 1 - Onboarding (fresh database)
**Database:** SQLite (fresh, no tables)
**Error:** `Table "dynamic_collections" not found in schema registry. Ensure setTableResolver() has been called during boot.`

**Root Cause:**
Phase 1 Task 6 removed the raw SQL fallback from the adapter's CRUD methods. Now ALL queries require a TableResolver (SchemaRegistry) to be set. But during boot, the code needs to query `dynamic_collections` to LOAD the collections BEFORE the SchemaRegistry is fully initialized. This is a chicken-and-egg problem:

1. Boot: create adapter
2. Boot: try to query dynamic_collections to load collections → FAILS (no TableResolver)
3. Boot: would have set up SchemaRegistry with loaded collections → never reaches here

**Pre-Phase 1 behavior:** The fallback raw SQL path handled this case. Queries to system tables (users, dynamic_collections) worked without a TableResolver because they used `buildSelectQuery()` directly.

**Impact:** Fresh databases cannot boot. `nextly dev --seed` fails. New projects created via CLI cannot start.

**Suggested Fix:**
The adapter needs to handle system table queries (dynamic_collections, users, etc.) WITHOUT requiring a TableResolver. Options:

1. **Restore fallback for system tables only** - If TableResolver is not set OR table not found in resolver, fall back to executeQuery for known system tables
2. **Register system tables in SchemaRegistry BEFORE querying** - The boot sequence should register static schemas first, then query dynamic_collections
3. **Use executeQuery directly for boot queries** - The boot sequence code in dev.ts should call adapter.executeQuery() with raw SQL for the initial dynamic_collections query

**Priority:** CRITICAL - blocks all e2e testing

## 2026-04-01 - Issue #5: Dynamic collection tables not in API route SchemaRegistry [CRITICAL]

**Test:** Test 3 - Content CRUD (create entry in Products collection)
**Database:** SQLite
**Error:** `Table "dc_product" not found in schema registry`

**Root Cause:**
The SchemaRegistry initialized in the DI container (register.ts) only loads static system tables via `getDialectTables()`. Dynamic collection tables (dc_product, dc_posts) are NOT loaded because they are created at runtime via the visual schema builder and their Drizzle table objects need to be generated from the `dynamic_collections` metadata.

The boot sequence in dev.ts handles this (loads from DB, generates runtime schemas, registers in registry), but the DI initialization in register.ts does NOT do this. API routes use the DI adapter which only has static tables.

**Impact:** Cannot create/read/update/delete entries in any dynamically-created collection via the admin UI. System table queries (users, permissions) work fine.

**Suggested Fix:**
In the DI register.ts, after registering static schemas, also:

1. Query `dynamic_collections` table for all registered collections
2. For each collection, call `generateRuntimeSchema()` to create Drizzle table objects
3. Register each in the SchemaRegistry via `registerDynamicSchema()`

This mirrors what the boot sequence does in dev.ts performAutoSync.

**Priority:** CRITICAL - blocks content CRUD for dynamic collections
