Nextly stores data through a small set of dialect-specific adapter packages built on a shared [Drizzle ORM](https://orm.drizzle.team/) core. Pick the adapter that matches your database, install it as a regular dependency, and Nextly handles dialect differences (placeholders, identifier escaping, RETURNING fallbacks, error classification) automatically.

## Supported databases

| Database | Package | Driver | Use it for |
|----------|---------|--------|------------|
| [PostgreSQL](/docs/database/postgres) | `@nextlyhq/adapter-postgres` | `pg` | **Recommended for production.** Full feature set: JSONB, arrays, full-text search, ILIKE, savepoints, RETURNING. |
| [MySQL](/docs/database/mysql) | `@nextlyhq/adapter-mysql` | `mysql2` | Supported. Use it when you already run MySQL or a MySQL-compatible cloud database. Some features are emulated -- see the page. |
| [SQLite](/docs/database/sqlite) | `@nextlyhq/adapter-sqlite` | `better-sqlite3` | **Local demos only.** Single-writer, file-based, no SSL. Not recommended for serious local development -- run Postgres in Docker instead. |

For minimum versions, cloud-provider compatibility, and the policy on MySQL-compatible variants (MariaDB, TiDB, Aurora MySQL, PlanetScale, Vitess), see [database support](/docs/database/support).

## Quick recommendation

> **Use PostgreSQL.** It is the recommended production database, the only adapter with the full feature set, and the easiest to run locally with Docker (`docker compose up -d postgres`). Pick MySQL only if you already run it. Pick SQLite only for one-off local demos.

## Installation

Each adapter is a regular npm package. Install the adapter and its peer driver in your project.

## Selecting a dialect

Nextly's runtime picks the dialect from environment variables, in this order:

1. **`DB_DIALECT`** -- explicit override. One of `postgresql`, `mysql`, `sqlite`.
2. **`DATABASE_URL`** protocol -- `postgres://` / `postgresql://` -> PostgreSQL, `mysql://` -> MySQL, `file:` (or `*.db` / `*.sqlite`) -> SQLite.
3. **Fallback** -- PostgreSQL with a warning. Always set `DB_DIALECT` explicitly to avoid the warning.

Minimal `.env` files for each dialect:

```bash title=".env (PostgreSQL)"
DB_DIALECT=postgresql
DATABASE_URL=postgres://nextly:nextly@localhost:5432/nextly_dev
```

```bash title=".env (MySQL)"
DB_DIALECT=mysql
DATABASE_URL=mysql://nextly:nextly@localhost:3306/nextly_dev
```

```bash title=".env (SQLite, local demo only)"
DB_DIALECT=sqlite
DATABASE_URL=file:./data/nextly.db
```

## Feature comparison

The adapters share a single CRUD API, but underlying dialect capabilities differ. The shared `DrizzleAdapter` base reports per-dialect capabilities at runtime; this table is a quick reference.

| Feature | PostgreSQL | MySQL | SQLite |
|---------|------------|-------|--------|
| JSONB columns | Yes | No (JSON only) | No (JSON as TEXT) |
| Array columns | Yes | No | No |
| `RETURNING` clause | Yes | No (extra `SELECT` after writes) | Yes (3.35+) |
| `ILIKE` (case-insensitive) | Native | `LOWER(...) LIKE LOWER(...)` fallback | `LOWER(...) LIKE LOWER(...)` fallback |
| Savepoints (nested transactions) | Yes | Disabled by adapter | Yes |
| Full-text search | `tsvector` | `FULLTEXT` indexes | FTS5 (extension) |
| Upsert syntax | `ON CONFLICT DO UPDATE` | `ON DUPLICATE KEY UPDATE` | `ON CONFLICT DO UPDATE` |
| Connection pooling | `pg.Pool` | `mysql2` pool | None (single file handle) |
| Generated columns | Yes | Yes | Yes |
| Max query parameters | 65,535 | 65,535 | 999 |
| SSL / TLS | Yes | Yes | N/A (local file) |
| Deadlock retry | On `40001` / `40P01` (configurable) | On error 1213 (configurable) | N/A |
| Auto-detected providers | Neon, Supabase, standard | None | None |

## How it works under the hood

All adapters extend `DrizzleAdapter` from `@nextlyhq/adapter-drizzle`. The base class provides:

- **CRUD** -- `select()`, `selectOne()`, `insert()`, `insertMany()`, `update()`, `delete()`, `upsert()` work identically across dialects.
- **Query building** -- `WHERE` / `ORDER BY` / `LIMIT` / joins emit dialect-aware SQL.
- **Migrations** -- a `nextly_migrations` table tracks applied files with SHA-256 checksums and per-migration status.
- **Error classification** -- driver-specific error codes are translated into a structured `DatabaseError` (`unique_violation`, `foreign_key_violation`, `timeout`, `serialization_failure`, `deadlock`, ...).
- **Transactions** -- automatic commit on success, rollback on error, optional retry for transient transaction failures.
- **Connect-time version check** -- on `connect()`, each adapter verifies the database is at or above the minimum version Nextly supports. Real dialects below minimum hard-fail; recognized MySQL-compatible variants warn and proceed; unparseable strings hard-fail. See [database support](/docs/database/support).

Each dialect adapter overrides only what differs. PostgreSQL uses `$1` placeholders and double-quoted identifiers; MySQL uses `?` placeholders and backticks; SQLite uses `?` placeholders and the same double-quoted identifiers as PostgreSQL.
