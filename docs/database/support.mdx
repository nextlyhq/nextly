---
title: Database support and version requirements
description: Minimum supported versions, cloud provider compatibility, and the warn-and-proceed policy for MySQL-compatible variants.
---

Nextly supports PostgreSQL, MySQL, and SQLite through dedicated adapter packages. Each connect attempt verifies the database version is at or above the minimum Nextly requires. Older real PostgreSQL/MySQL/SQLite versions are refused at boot with a clear upgrade pointer; recognized cloud-flavored MySQL variants are allowed to proceed with a one-time warning.

## Minimum versions

| Database | Minimum | Why this minimum |
|----------|---------|------------------|
| PostgreSQL | 15.0 | Native MERGE statement support; modern btree dedup; full transactional DDL parity. PG 15 is the oldest version still in upstream support. |
| MySQL | 8.0 | Native `RENAME COLUMN` (8.0); replaces MySQL 5.7's `CHANGE COLUMN` recreate dance. 8.0 is the only modern major in active support. |
| SQLite | 3.38 | Native `unixepoch()`, strict tables, and improved query planner. 3.38 ships with most distributions Nextly's pinned `better-sqlite3` supports. |

If your database reports a version below the minimum, the adapter's `connect()` call throws `UnsupportedDialectVersionError` with a message pointing at this page.

## Cloud provider compatibility

### PostgreSQL — supported

| Provider | Status | Notes |
|----------|--------|-------|
| Standard PostgreSQL (self-hosted, Docker, RDS, Cloud SQL) | Supported | All versions 15.0+ work without configuration. |
| [Neon](https://neon.tech) | Supported | Auto-detected from URL pattern. Cold-start retries built into the adapter. |
| [Supabase](https://supabase.com) | Supported | Auto-detected. SSL enforced. |
| AWS RDS / Aurora PostgreSQL (PG-flavored Aurora, not MySQL Aurora) | Supported | RDS reports as standard PostgreSQL; Aurora PG also passes the version check on the PG-compatible majors. |

### MySQL — supported with warnings for variants

Nextly v1 is regression-tested against **real MySQL 8.0+** only. MySQL-protocol-compatible variants are recognized at connect, log a one-time warning through your adapter's logger, and proceed:

| Variant | Behavior |
|---------|----------|
| **MariaDB** (any version) | Warning + proceed. Wire-compatible with MySQL; most operations work but feature drift is possible on advanced SQL. |
| **TiDB** | Warning + proceed. Advertises MySQL 5.7 protocol; modern TiDB versions support enough MySQL 8 features for typical CMS workloads. |
| **AWS Aurora MySQL** | Warning + proceed. Internal version reporting differs from upstream MySQL. |
| **PlanetScale / Vitess** | Warning + proceed. Vitess-based MySQL-compatible layer. |

The warning text routes through the adapter's configured logger and looks like this:

```
Detected mariadb ('10.11.5-MariaDB-...'). Nextly is regression-tested
against real MySQL 8.0+ only; this MySQL-compatible variant is not
officially supported in v1 but most operations should work. See
https://nextlyhq.com/docs/database/support.
```

If you hit a variant-specific bug, please file an issue with your variant + version. v2 will add opt-in regression coverage for individual variants.

If a MySQL response is **completely unparseable** (no version number found AND no variant token detected), Nextly hard-fails so the issue surfaces at boot rather than mid-apply later.

### SQLite — supported

| Provider | Status | Notes |
|----------|--------|-------|
| `better-sqlite3` (default) | Supported | Bundled SQLite is whatever ships with the pinned `better-sqlite3` version (currently 3.45+). |
| `bun:sqlite` | Not yet supported | Adapter is in scaffold state. |
| `libSQL` / Turso | Not yet supported | Adapter is in scaffold state. |

## Upgrade guidance

### PostgreSQL

If you are on PG 12, 13, or 14: most managed providers offer in-place major-version upgrades. Run `SELECT version()` to confirm; consult your provider's docs for upgrade procedure.

### MySQL

If you are on MySQL 5.7: upstream support ended in October 2023. Your provider almost certainly offers a managed upgrade to 8.0. Test the upgrade in a staging environment first since some 5.7 SQL is incompatible with 8.0 reserved-word changes.

### SQLite

Bundled with `better-sqlite3` — upgrade by bumping the package version. If you maintain your own SQLite linkage, install at least 3.38 from your distribution's package manager.

## Why the policy is hybrid

Nextly draws three different lines at connect time:

1. **Real PG/MySQL/SQLite below minimum → hard-fail.** Schema migrations rely on dialect features at the minimum-version threshold (RENAME COLUMN, transactional DDL, MERGE, strict tables). A real MySQL 5.7 user who proceeded would hit confusing mid-apply errors during their first schema change.
2. **Recognized variant (MariaDB, TiDB, Aurora MySQL, PlanetScale, Vitess) → warn and proceed.** These are real cloud databases millions of developers run. Hard-failing them on first install would block legitimate users from a CMS that mostly works on their stack. The warning makes the trade-off explicit.
3. **Completely unparseable string → hard-fail.** If we cannot find a version number AND cannot detect a variant token, we surface the issue immediately rather than silently letting the user proceed and hit cryptic errors later.

The error and warning paths both include a link to this page so users know the upgrade lever or, for variants, that they are running on best-effort support.

## When the version check runs

`checkDialectVersion()` runs inside each adapter's `connect()` method, after the connection is verified alive but before the adapter marks itself ready. PostgreSQL retries transient network failures (Neon cold starts, `EAI_AGAIN`) without re-running the version query in a tight loop; only confirmed version mismatches throw `UnsupportedDialectVersionError`.

## Catching the error

The error implements adapter-drizzle's `DatabaseError` interface, so existing `isDatabaseError()` type guards continue to work. You can also branch on the `kind` field directly:

```ts
import { UnsupportedDialectVersionError } from "@nextlyhq/adapter-drizzle/version-check";

try {
  await adapter.connect();
} catch (error) {
  if (error instanceof UnsupportedDialectVersionError) {
    console.error(
      `Detected ${error.detectedVersion}, required ${error.requiredVersion}`
    );
    process.exit(1);
  }
  throw error;
}
```
